Independent candidate vision
PAGE 1 · DAYS 0—60

Prove one correction loop before scaling the system.

The first two passes create a shared correction contract, replayable evidence, and a narrow shadow loop. The goal is to reduce uncertainty without using the machine floor as an avoidable test harness.

DAYS 0—15

Select the first correction class

Choose a problem with customer value, measurable outcomes, sufficient evidence, bounded consequence, and platform leverage.

Exit: named owner, target state, success criteria, hold conditions.
DAYS 16—30

Build the correction contract

Define intent, state, coordinate frames, data sources, cause hypotheses, action limits, authority, validation, and reuse scope.

Exit: replayable baseline and signed correction object.
DAYS 31—45

Build the evidence harness

Create deterministic replay, simulation inputs, state visualization, experiment tracking, and a representative regression set.

Exit: repeatable test path before machine action.
DAYS 46—60

Close the shadow loop

Reconstruct state, rank causes, propose bounded corrections, compare with expert action, and plan post-action proof.

Exit: disagreement log, rollback, before/after evidence plan.

Independent candidate vision
PAGE 2 · DAYS 61—120

Stress the boundary, then install the compounding system.

DAYS 61—75

Stress the operating envelope

Test warm-up, environment, tool and material variation, fixture and coordinate changes, program style, missing data, sensor degradation, and rollback.

Exit: failure taxonomy and known limits.
DAYS 76—90

Qualify the release posture

Compare shadow recommendations with expert decisions, establish post-correction proof, and decide whether the class remains shadow, earns bounded release, or holds.

Exit: explicit release and incident rules.
DAYS 91—105

Make architecture boundaries durable

Install data and frame contracts, model registry, simulation interface, evaluation, observability, lineage, retention, and rollback.

Exit: first proven platform contracts.
DAYS 106—120

Build the founding AI operating cadence

Clarify ownership, hiring priorities, experiment review, correction review, architecture decisions, product evidence, and correction-class portfolio choices.

Exit: team design and next two correction classes.
